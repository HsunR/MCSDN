<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → PublicCommentController | Untrusted input (authorName, content) crosses here via @RequestBody |
| client → AdminCommentController | JWT token required, but user-controlled content still crosses |
| PublicCommentController → CommentService | Internal call, validated input |
| CommentService → CommentMapper | SQL parameter binding via #{ } |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | CommentMapper.xml | mitigate | All user inputs use MyBatis `#{ }` binding — never `${}`. Verified in every SQL statement. |
| T-03-02 | XSS | Comment.content | mitigate | Vue `{{ }}` text interpolation in frontend escapes HTML. Backend stores raw text. No v-html on user content. |
| T-03-03 | Denial of Service | PublicCommentController | mitigate | Honeypot field (name="website") catches bots. SpamException returns 200 with success message — no error signal to bots. |
| T-03-04 | Info Disclosure | PublicCommentController | mitigate | GET returns only `getApprovedComments()` — PENDING comments never exposed publicly. |
| T-03-05 | Tampering | AdminCommentController | accept | Admin endpoints require JWT via existing /api/admin/** rule. IDOR risk is low — admin manages their own article comments. |
</threat_model>
